# Prompt：从功能描述与代码改动生成 PRD 草稿

> **如何使用**：把整篇内容发给 LLM，附上功能简述、关键代码改动、相关 API 与数据模型。LLM 将输出可直接保存到 `docs/requirements/REQ-<feature>.md` 的 PRD 草稿。

---

## 角色

你是一名经验丰富的产品经理，擅长把已落地的代码逆向整理成清晰的需求文档（PRD），让没有看过代码的同事也能理解功能。

## 上下文

**功能短名**（用于文件名，如 `user-auth`）：<FEATURE_NAME>

**一句话描述**（必填）：<ONE_LINE_DESCRIPTION>

**业务背景**（必填）：
<为什么要做这个功能？解决谁的什么问题？数据来源/用户反馈/竞品分析…>

**关键代码改动**（必填）：
```<语言>
<把核心改动贴这里，包括：新增/修改的接口、关键算法、数据模型变更>
```

**已有 API 接口**（可选）：
<已经存在的相关接口，用 `METHOD /path` 列表表示>

**相关数据表**（可选）：
<表名、关键字段；可用 Mermaid ER 图>

**目标用户**（可选但推荐）：<例如：注册用户、商家、内部运营>

**期望上线日期**（可选）：<YYYY-MM-DD>

## 任务

输出一份可直接保存到 `docs/requirements/REQ-<FEATURE_NAME>.md` 的 Markdown，结构遵循项目 `templates/PRD.md`。

## 必须遵循的规则

1. **章节结构与 `templates/PRD.md` 完全一致**：
   1. 文档信息（编号、版本、日期、作者、状态、优先级）
   2. 功能概述（简要描述、关键词、范围）
   3. 背景和目标（背景、SMART 目标、非目标、成功指标）
   4. 功能需求（用户故事、功能清单、业务规则、边界与异常）
   5. 非功能需求（性能、安全、兼容性、可观测性）
   6. UI / 交互设计（如非纯后端功能）
   7. 数据模型
   8. 接口设计（链接到 API.md，列出接口清单）
   9. 验收标准（功能验收、测试用例、性能验收）
   10. 上线计划（里程碑、灰度策略、回滚方案）
   11. 风险与依赖
   12. 附录（相关文档、变更历史）
2. **基于代码反推**：
   - 用户故事必须能在代码中找到对应实现
   - 业务规则要从代码逻辑（条件分支、校验、状态机）中提炼
   - 接口清单要与实际接口完全一致
3. **不要编造非功能指标**：性能、可观测性等没有依据时，写「待业务方确认」或「沿用项目默认 SLO」。
4. **明确范围**：把代码中**未实现**但容易混淆的功能写入「2.3 范围说明 - 不包含」。
5. **可测试的验收标准**：每条 AC 必须能被一个具体的测试步骤验证。
6. **文档信息**：
   - 编号：`REQ-XXX`（用 `???` 占位，由 update_docs.py req 命令分配）
   - 状态：`已实现`（因为是逆向整理）
   - 创建日期：今天
7. **保留模板里的 Mermaid ER 图骨架**（即便简单）。
8. **中文撰写**。

## 输出格式

直接输出完整的 Markdown，从 `# <功能标题> - 需求文档` 开始，到附录结束。不要输出额外说明。
